# UI Implementation Summary

## 1. Purpose

The web UI was added to provide an interactive demo layer over the existing REST API, enabling non-technical stakeholders and reviewers to exercise the system without `curl` or Swagger.

The motivation is purely demonstrational:

- Reduce friction when showcasing the classification and retrieval pipeline.
- Present a self-contained artifact that runs with a single `docker compose up --build`.
- Keep the UI strictly decoupled from the API — it consumes `/predict` as any external client would.

The separation is explicit: the UI is a static HTML file served by the same FastAPI process. It introduces no coupling to business logic, no shared state, and no new service boundary.

---

## 2. Architecture Overview

```
Browser
  |
  |  GET /          → FastAPI serves app/templates/index.html (FileResponse)
  |  POST /predict  → FastAPI routes to classifier + RAG + LLM mock
  |
FastAPI (app/main.py)
  ├── GET  /          FileResponse — static HTML
  ├── POST /predict   unchanged
  └── GET  /health    unchanged
```

Key design decisions:

- **No external frontend framework.** The UI is a single HTML file with an inline `<style>` block and a `<script>` block using the native browser `fetch()` API.
- **No template engine.** The file is served with `fastapi.responses.FileResponse`. Jinja2 was not added and is not required.
- **No new service.** The UI runs inside the same container and process as the API. No nginx, no separate port, no reverse proxy.
- **No new Python dependency.** `FileResponse` ships with FastAPI. `requirements.txt` is unchanged.

## 4. How It Works

### Request flow

1. Browser sends `GET /` to FastAPI.
2. FastAPI returns `app/templates/index.html` via `FileResponse` (HTTP 200, `text/html`).
3. User types a ticket description into the `<textarea>` and clicks **Run Prediction** (or presses `Ctrl+Enter`).
4. The inline JavaScript `fetch()` call sends `POST /predict` with `Content-Type: application/json` and a body of `{"ticket_text": "<input>"}`.
5. The browser receives the JSON response from the unchanged `/predict` handler.
6. The script renders the full JSON response in a `<pre>` block and displays three summary badges: category, urgency level, and latency.

### Data path (unchanged)

```
POST /predict
  → preprocessing.clean_text()
  → classifier.predict()        (scikit-learn TF-IDF + Logistic Regression)
  → retriever.retrieve()        (ChromaDB vector search)
  → llm.generate_response()     (template-based mock, no external API)
  → TicketResponse JSON
```

## 5. What It Is Not

| Aspect | Status |
|---|---|
| Production UI | No. This is a demo interface only. |
| Authenticated | No. There is no login, token, or session. |
| Stateful | No. No history is stored; each request is independent. |
| Connected to an external LLM | No. The response is generated by a local template function (`app/services/llm.py`). |
| A standalone frontend service | No. It is served by the same FastAPI process; there is no separate build step or bundle. |
| Tested by automated tests | No. The HTML and JS are not covered by the pytest suite. `/predict` and `/health` are. |

## 6. How to Run

### Docker (recommended)

```bash
# First-time setup
copy .env.example .env        # Windows
# cp .env.example .env        # Linux / macOS

docker compose up --build
```

Open a browser and navigate to:

| URL | Description |
|---|---|
| `http://localhost:8000/` | Demo web UI |
| `http://localhost:8000/docs` | Swagger UI (auto-generated by FastAPI) |
| `http://localhost:8000/health` | Health check endpoint |

### Local (Python)

After completing the local setup steps in `RUN.md`:

```bash
uvicorn app.main:app --host 0.0.0.0 --port 8000
```

Then open `http://localhost:8000/` in a browser.

## 7. Design Principles

**Minimal.** The implementation adds one HTML file and four lines to `main.py`. Nothing else changes.

**No new dependencies.** `FileResponse` is part of the FastAPI standard library. The Python environment is identical before and after the UI was added.

**Core API untouched.** The behavior of `POST /predict` and `GET /health` is unchanged in every respect — routing, validation, response schema, logging, and latency characteristics.

**Demo-focused.** The UI prioritises clarity of output (pretty-printed JSON, summary badges, latency display) over visual polish. It is not designed for real-world usage patterns such as concurrent users, accessibility compliance, or mobile layout.

**Self-contained.** The HTML file has no external CSS or JS dependencies. It works offline once the container is running.
